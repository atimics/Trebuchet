# Signal↔Solana Wallet Bridge

## The problem

A player in Signal needs to burn RATi tokens at a furnace. The burn happens on
Solana. The ingot creation happens in Signal's game sim. The bridge connects
two identities: the player's Signal keypair (which signs game actions) and
their Solana wallet (which holds RATi and signs burn transactions).

The question: how does the Signal server know which Solana wallet belongs to
which player?

## Design space

### Option A: Same keypair

Player uses one Ed25519 keypair for both Signal and Solana. No bridge needed.
The pubkey is the identity in both systems.

**Rejected.** Signal identity keys are generated by the game client on first
launch. Solana wallets are managed by Phantom, Backpack, or hardware wallets.
Most players arrive with a Solana wallet already. Sharing a key between a game
and a wallet is bad security — a compromised game client leaks the wallet.

### Option B: Signed association

Player proves control of both keypairs by signing a link message with each.
The Signal server stores the association. This is the standard "link wallet"
pattern used by most Web3 games.

**Works, but adds a step.** The player must sign two messages before their
first furnace smelt. After that, the link is persistent.

### Option C: Same BIP39 seed, different derivation paths

Signal generates a BIP39 mnemonic on first launch. The Signal game key is
derived at `m/44'/501'/0'/0'` (or a custom path). The player imports the
same mnemonic into their Solana wallet. Same seed, different keys.

**Rejected.** Requires players to import a game-generated mnemonic into their
wallet. Most players want the opposite: bring their existing wallet to the
game. Also, Signal currently uses raw Ed25519 keypairs, not BIP39.

## Chosen design: Option B with extensions

The player links their Solana wallet to their Signal identity once. After
linking, the Signal server can map game actions to Solana state.

### The link ceremony

```
PLAYER                          SIGNAL SERVER                  SOLANA
──────                          ─────────────                  ──────

1. Request link
   ├─ signal_pubkey
   └──────────────────────────► 2. Generate link challenge:
                                   challenge = SHA256(
                                     "signal-wallet-link-v1" ||
                                     signal_pubkey ||
                                     server_nonce
                                   )

                                3. Return challenge + nonce
   ◄──────────────────────────

4. Sign challenge with
   Signal identity key
   ├─ sig_signal = Ed25519.sign(
   │     identity_secret,
   │     challenge
   │   )
   └──────────────────────────► 5. Verify sig_signal against
                                   signal_pubkey ✓

                                6. Return Solana link message:
                                   link_msg = "signal-wallet-link-v1" ||
                                              signal_pubkey || solana_pubkey

   ◄──────────────────────────

7. Sign link message with
   Solana wallet
   ├─ sig_solana = Ed25519.sign(
   │     solana_secret,
   │     link_msg
   │   )
   └──────────────────────────► 8. Verify sig_solana against
                                   solana_pubkey ✓

                                9. Store link:
                                   signal_pubkey → solana_pubkey
                                   (persisted across restarts)

                                10. Confirm link
   ◄──────────────────────────
```

After the ceremony, the Signal server knows that `signal_pubkey` controls
`solana_pubkey`. The link is stored in the player's save file and in the
server's identity registry.

### Link storage

The link is stored in two places:

1. **Player save file** (`saves/pubkey/<base58(signal_pubkey)>.sav`) —
   `player_t.linked_solana_pubkey[32]`. Survives server restart.

2. **Server identity registry** — an in-memory map of `signal_pubkey →
   solana_pubkey`. Rebuilt from save files on startup. Used for quick
   lookups during furnace operations.

The link can be verified at any time: the server holds both signatures from
the ceremony. A third party can verify the link without trusting the server
by checking both signatures against the link message.

### Multiple wallets

A player MAY link multiple Solana wallets to their Signal identity. The
furnace UI lets them choose which wallet to burn from. The server tracks
all linked wallets:

```
player_t.linked_solana_wallets[] — array of up to 8 pubkeys
player_t.linked_solana_proofs[]  — corresponding signatures
```

This supports: a hot wallet for small burns, a cold wallet for large burns,
a mobile wallet for convenience.

## The furnace smelt flow (end to end)

With the wallet link established, here is the complete smelt flow:

```
PLAYER (Signal client)          SIGNAL SERVER                  SOLANA
──────────────────────          ─────────────                  ──────

1. Dock at station with
   furnace module

2. Open furnace UI
   ├─ Select linked wallet
   ├─ Enter RATi amount to burn
   └──────────────────────────► 3. Validate:
                                   - Player linked to wallet? ✓
                                   - Wallet has ≥ amount RATi?
                                   - Station has furnace module?
                                   - Furnace not busy?

                                4. Build burn transaction:
                                   ix = burn(
                                     token_account,
                                     amount,
                                     furnace_pda
                                   )
                                   tx = build_and_sign_locally?

                                5. Return unsigned tx
   ◄──────────────────────────    (or: server builds, player signs)

6. Player signs tx with
   Solana wallet (Phantom)
   └────────────────────────────────────────────────────────► 7. Submit to Solana
                                                                  RPC

                                                                8. Burn confirmed
                                                                   tx_sig = "abc..."

                                                                9. RATi supply decreases

                                10. Server polls Solana:
                                    - Find burn tx from linked wallet
                                    - Verify: correct amount, correct
                                      furnace program, correct station

                                11. Create ingot:
                                    cargo_unit.pub = hash(
                                      FURNACE_COMMODITY,
                                      grade,
                                      burn_tx_sig,  ← Solana tx
                                      output_index
                                    )
                                    parent_merkle = burn_tx_sig

                                12. Record smelt event:
                                    CHAIN_EVT_FURNACE_SMELT
                                    signed by station authority
                                    payload: cargo_unit.pub,
                                             burn_tx_sig,
                                             amount_burned,
                                             player_pubkey

                                13. Add ingot to player manifest
                                    → player now has ingot in-game

   ◄────────────────────────── 14. Furnace UI: show ingot
                                    with provenance link to
                                    Solana burn tx
```

### Who builds the transaction?

Two models, both valid:

**Model 1: Server builds, player signs in Phantom.**
The Signal server constructs the Solana burn transaction and returns it to
the client. The client hands it to Phantom for signing. Phantom submits to
Solana RPC. This is the standard dApp pattern. The server doesn't hold
the player's Solana key.

**Model 2: Trebuchet CLI builds, player signs.**
The packnft CLI (Signal `tools/packnft/`) builds the burn transaction.
The client signs with Phantom. This is cleaner — no Solana transaction
building in the game server — but requires packnft to be available.

**Recommendation: Model 2.** The packnft CLI already handles transaction
building for NFTs. Adding a `build-furnace-burn` operation to packnft is
a small addition. The game server stays focused on simulation. Packnft
owns all Solana transaction construction.

### Ingot identity from burn transaction

The ingot's content address is derived from the Solana burn transaction:

```c
cargo_unit.pub = hash_furnace_ingot(
    commodity,     // FURNACE_COMMODITY
    grade,         // based on furnace tier
    burn_tx_sig,   // 32-byte Solana transaction signature
    burn_amount,   // how much RATi was burned
    output_index   // which ingot from this burn (usually 0, but
                   // large burns could produce multiple ingots)
)
```

This means:
- The ingot is uniquely identified by the burn transaction
- The provenance chain is: rock → fragment → burn-to-mint RATi → furnace burn → ingot
- Anyone can verify the ingot by looking up the Solana burn tx
- The burn tx proves exactly how much RATi was consumed

### What the Signal server needs to read from Solana

The server needs a Solana RPC connection for:

1. **Balance checks:** Does the player's linked wallet have enough RATi?
2. **Burn verification:** Did the burn transaction confirm? What amount was burned?
3. **Token account discovery:** Which token accounts does the wallet have?
4. **Transaction polling:** Wait for burn confirmation (typically 1-2 slots)

This is read-only RPC access. The server never signs Solana transactions.
It reads state to verify player actions.

### Server-side Solana client

```c
// server/solana_bridge.h — new file

// Check if a wallet has at least `amount` RATi
bool solana_bridge_check_balance(
    const char *rpc_url,
    const uint8_t wallet_pubkey[32],
    const uint8_t rati_mint[32],
    uint64_t amount
);

// Verify a burn transaction
bool solana_bridge_verify_burn(
    const char *rpc_url,
    const char *tx_signature_b58,
    const uint8_t expected_wallet[32],
    const uint8_t expected_mint[32],
    uint64_t expected_amount,
    uint8_t actual_amount_out[8]  // what was actually burned
);

// Find token accounts for a wallet
int solana_bridge_find_token_accounts(
    const char *rpc_url,
    const uint8_t wallet_pubkey[32],
    const uint8_t mint[32],
    uint8_t token_accounts_out[8][32],  // up to 8
    uint64_t balances_out[8]
);
```

## Security considerations

### Replay attacks

A burn transaction on Solana is final. The Signal server must ensure it
doesn't process the same burn twice. The ingot's `cargo_unit.pub` includes
the burn tx signature, which is globally unique. Attempting to create a
second ingot from the same burn tx would produce the same `cargo_unit.pub`,
and the manifest rejects duplicates.

### Front-running

A malicious player could submit their own burn transaction before the
server builds one. This is fine — the server is verifying burns, not
authorizing them. If the burn tx is valid (correct amount, correct
furnace program, linked wallet), the server accepts it. The player
can't "trick" the server into creating an ingot from a burn that didn't
happen.

### Wallet unlinking

If a player unlinks a wallet, previously verified burns are not affected.
The ingots already exist. The player simply can't use that wallet for
future furnace operations until they re-link.

### Station authority

The furnace smelt event is signed by the station's Ed25519 keypair
(`CHAIN_EVT_FURNACE_SMELT`). This proves which station performed the
smelt and at what sim epoch. A compromised game server could forge smelt
events — but the station's chain log would show the forgery, and the
Solana burn transaction provides an independent on-chain record that
can be cross-referenced.

## What this means for #480

#480 (on-chain anchoring of Signal chain-tip hashes) makes the bridge
stronger but is not required for the bridge to work. Without #480:

- The furnace flow works as described: player burns RATi on Solana, server
  verifies, creates ingot.
- The ingot's provenance stops at the Solana burn tx. You can verify the
  burn happened, but you can't verify the Signal server's smelt event
  without trusting the server (or its chain log).

With #480:

- The station's chain-tip hash is anchored to Solana.
- The `CHAIN_EVT_FURNACE_SMELT` event is part of the station's signed
  chain log.
- Anyone can verify: burn happened (Solana) → smelt event recorded
  (chain log, anchored on Solana) → ingot created (cargo_unit.pub derived
  from burn tx).
- The full provenance is auditable without trusting the game server.

#480 closes the trust gap but doesn't block the initial implementation.

## Implementation path

1. **Player identity → Solana wallet linking.** Add `linked_solana_pubkey`
   to `player_t` in `shared/types.h`. Add link ceremony to client and
   server. Store link in save file.

2. **Solana RPC client in Signal server.** Add `server/solana_bridge.c`
   with balance checks and burn verification. HTTP client for Solana
   JSON-RPC (Signal already has HTTP for other purposes, or add a minimal
   one).

3. **Furnace module UI.** Add furnace tab to station UI in `client/`.
   Show linked wallets, RATi balance, burn amount input, "Smelt" button.

4. **Burn transaction in packnft CLI.** Add `build-furnace-burn` operation
   to `signal/tools/packnft/cli.c`. Builds a signed SPL Token burn
   transaction.

5. **Ingot creation in game sim.** Add `sim_furnace_smelt()` to
   `server/sim_production.c`. Reads burn verification from Solana,
   computes `cargo_unit.pub`, adds ingot to player manifest, emits
   `CHAIN_EVT_FURNACE_SMELT`.

6. **#480 on-chain anchoring.** Makes the full provenance chain verifiable
   without trusting the game server.
